\section{Introducción}
\label{sec:intro}

Este informe documenta de forma técnica exhaustiva las dos primeras fases del
trabajo de fin de estudios (TFE):

\begin{itemize}
  \item \textbf{Fase~1} establece líneas base de eficiencia y calidad sobre
    tres modelos \textbf{Qwen2.5-Instruct} (14B, 7B y 1.5B) desplegados en
    MareNostrum~5 mediante vLLM. Estas líneas base cuantifican tanto el
    rendimiento de servicio (throughput y latencia) como la calidad de respuesta
    (benchmarks GSM8K y MATH-500) de los modelos sin ninguna
    adaptación posterior.

  \item \textbf{Fase~2} aplica \emph{Knowledge Distillation} black-box: el
    teacher Qwen2.5-14B genera 1819 respuestas greedy sobre los mismos benchmarks;
    esas respuestas se usan como señal de entrenamiento para ajustar finamente
    (SFT) a los dos estudiantes (7B y 1.5B) con adaptadores LoRA, y a
    continuación se re-evalúan en los mismos benchmarks para cuantificar la
    ganancia de calidad atribuible a la destilación.
\end{itemize}

El objetivo final es demostrar que un estudiante destilado puede recuperar una
parte sustancial de la calidad del teacher a una fracción de su coste de servicio,
habilitando despliegues eficientes en entornos de supercomputación con restricciones
de memoria y tiempo de cómputo.

\section{Estructura del Proyecto y Arquitectura del Código}
\label{sec:architecture-code}

El repositorio está organizado en módulos Python bien delimitados, scripts SLURM
independientes por tarea, y ficheros de configuración YAML que parametrizan cada
experimento. Esta organización permite reutilizar módulos (por ejemplo, el scoring
es compartido por todos los benchmarks) y reproducir cualquier resultado con un
simple \texttt{sbatch}.

\subsection{Sistema de reproducibilidad (\texttt{utils/reproducibility.py})}

Toda ejecución crea un \textbf{directorio de run} con timestamp UTC bajo
\texttt{results/<subsistema>/}, mediante \texttt{make\_run\_dir(base, tag)}.
La función \texttt{snapshot\_configs} copia todos los YAMLs activos al directorio
de run.  \texttt{collect\_metadata} registra en \texttt{metadata.json}: hostname,
nombre y memoria de GPU, versión de CUDA, versión de Python, git SHA, semilla
aleatoria y hashes de los ficheros de config.  \texttt{set\_seed(seed)} fija las
semillas de Python \texttt{random}, NumPy y PyTorch (incluyendo
\texttt{cuda.manual\_seed\_all}).

\section{Fase 1A — Líneas Base de Eficiencia}
\label{sec:efficiency}

\subsection{Metodología}

La eficiencia se mide en dos ejes ortogonales que cubren los dos modos
principales de uso de LLMs en producción:

\begin{enumerate}[leftmargin=2em]
  \item \textbf{Rendimiento máximo (offline):}
    Se envía un lote sostenido de 1000 peticiones ($\sim$150 tokens de entrada,
    256 tokens de salida cada una) tan rápido como sea posible (\texttt{request\_rate=inf}).
    La métrica es \emph{peticiones completadas por segundo} ($\text{req}/\text{s}$).
    Esto simula el caso de uso de procesamiento por lotes donde la latencia
    por petición individual no importa pero la productividad total es crítica.

  \item \textbf{Latencia online bajo carga:}
    Un flujo de peticiones distribuido según un proceso de Poisson a tasa
    $\lambda = 10\,\text{req/s}$ se sostiene durante 120 segundos. Las métricas
    reportadas son percentiles de latencia end-to-end (p50, p95, p99) y
    time-to-first-byte (TTFB p50). Esto simula el caso de uso interactivo con
    usuarios concurrentes.
\end{enumerate}

Todos los experimentos usan un servidor vLLM dedicado en un nodo H100 exclusivo.

\subsection{Módulo \texttt{bench/run\_throughput.py}}

Este script implementa el benchmark de rendimiento máximo invocando la herramienta
oficial \texttt{vllm bench serve} como subproceso. La función \texttt{\_build\_bench\_cmd}
construye la invocación de CLI con los parámetros leídos de \texttt{configs/benchmark.yaml}:

\begin{lstlisting}[language=Python, caption={Construcción del comando vllm bench serve.}]
cmd = [
    "vllm", "bench", "serve",
    "--base-url", "http://localhost:8000",
    "--model",    model_id,
    "--dataset-name", "sharegpt",
    "--num-prompts",  str(1000),      # 1000 peticiones en lote
    "--request-rate", "inf",          # tan rapido como sea posible
    "--result-filename", str(result_file),
]
\end{lstlisting}

\noindent El script crea un directorio de run con timestamp
(\texttt{results/throughput/throughput-<role>-<ts>/}), ejecuta el comando,
parsea la salida JSON y guarda las métricas en JSON y CSV junto al snapshot
de la configuración activa. La función \texttt{parse\_json\_result} del módulo
\texttt{bench/parse\_results.py} extrae el campo \texttt{request\_throughput} (req/s)
y otros campos del fichero de resultados generado por vLLM.

\subsection{Módulo \texttt{bench/run\_online\_load.py}}

El benchmark de carga online implementa un \textbf{cliente HTTP asíncrono}
basado en \texttt{httpx.AsyncClient} con llegadas Poisson. La función
\texttt{\_send\_request} mide con precisión de nanosegundo el TTFB (primer
chunk recibido) y la latencia end-to-end vía streaming:

\begin{lstlisting}[language=Python, caption={Medición de TTFB y latencia en run\_online\_load.py.}]
t_start = time.perf_counter()
ttfb: Optional[float] = None

async with client.stream("POST", url, json=payload) as resp:
    async for chunk in resp.aiter_bytes():
        if ttfb is None:         # primer byte recibido
            ttfb = (time.perf_counter() - t_start) * 1000.0  # ms
        chunks.append(chunk)

latency_ms = (time.perf_counter() - t_start) * 1000.0
\end{lstlisting}

\noindent Las llegadas se generan con distribución exponencial
(\texttt{numpy.random.exponential(1/lambda)}) para modelar un proceso de Poisson
con $\lambda = 10\,\text{req/s}$. Los percentiles se calculan mediante
\texttt{bench/metrics.py::summarise\_latencies} usando \texttt{numpy.percentile}.

\subsection{Jobs SLURM de eficiencia}

Cada modelo se mide con dos jobs SLURM:
\begin{itemize}
  \item \texttt{slurm/bench\_throughput.sbatch}: lanza el servidor vLLM, espera
    \texttt{/health}, ejecuta \texttt{bench/run\_throughput.py}, detiene el servidor.
    Tiempo asignado: 2h, 1$\times$H100, nodo exclusivo.
  \item \texttt{slurm/online\_load.sbatch}: lanza el servidor, espera \texttt{/health},
    ejecuta \texttt{bench/run\_online\_load.py} a $\lambda=10$, detiene el servidor.
    Tiempo asignado: 1h, 1$\times$H100, nodo exclusivo.
\end{itemize}

\noindent La exclusividad del nodo (\texttt{--exclusive}) garantiza ausencia de
interferencias de otros procesos sobre el acelerador, esencial para la
reproducibilidad de los benchmarks de latencia.

\subsection{Configuración del servidor vLLM}

\begin{itemize}[leftmargin=2em]
  \item \texttt{max\_model\_len = 4096}
  \item \texttt{gpu\_memory\_utilization = 0.90}
  \item \texttt{max\_num\_seqs = 256} (concurrencia máxima de secuencias en vuelo)
  \item \texttt{tensor\_parallel\_size = 1} (un solo GPU)
  \item \texttt{dtype = auto} (bfloat16 en H100)
\end{itemize}

\subsection{Conclusiones — Eficiencia}

\begin{itemize}
  \item El estudiante de 7B ofrece \textbf{$2.15\times$} mayor rendimiento y
    \textbf{$2.02\times$} menor latencia que el teacher.
  \item El estudiante de 1.5B logra \textbf{$4.14\times$} mayor rendimiento y
    \textbf{$6.71\times$} menor latencia.
  \item Ambos estudiantes caben cómodamente en una sola H100 con alta
    utilización de memoria GPU ($\geq 0.90$), facilitando el despliegue eficiente.
  \item La brecha de eficiencia motiva la destilación de conocimiento: si se
    puede recuperar la mayor parte de la calidad del teacher en un modelo más
    pequeño, la mejora en el coste de servicio es dramática.
\end{itemize}

\end{itemize}

\subsection{Orquestador unificado — \texttt{eval/run\_quality.py}}
\label{sec:eval-methodology}

El script \texttt{eval/run\_quality.py} es el punto de entrada único para todas
las evaluaciones de calidad. Su función \texttt{run(config\_path, model\_override,
role)} realiza los siguientes pasos:

\begin{enumerate}
  \item Carga \texttt{configs/eval.yaml} y aplica cualquier override de modelo.
  \item Crea un directorio de run con timestamp
    (\texttt{results/quality/quality-<role>-<ts>/}).
  \item Hace snapshot de todos los YAMLs activos y registra metadata de hardware.
  \item Para cada benchmark habilitado en \texttt{benchmarks:}, llama al módulo
    específico (\texttt{eval.gsm8k}, \texttt{eval.math\_eval},
    \texttt{eval.math\_eval}) y acumula las métricas.
  \item Serializa el resumen agregado en \texttt{quality\_summary.json} y
    \texttt{quality\_summary.csv}.
\end{enumerate}

\paragraph{Parámetros de generación.}
Todas las evaluaciones usan \textbf{decodificación greedy} (\texttt{temperature=0.0},
\texttt{top\_p=1.0}) con \texttt{max\_tokens=1024}, garantizando salidas
determinísticas y reproducibles. La semilla aleatoria se fija en 42.

\paragraph{Jobs SLURM de calidad.}
Los scripts \texttt{slurm/eval\_quality.sbatch}, \texttt{slurm/eval\_student\_mid.sbatch}
y \texttt{slurm/eval\_student\_small.sbatch} siguen el mismo patrón de tres etapas:
(1) arrancar servidor vLLM en GPU exclusiva, (2) health check con hasta 10 minutos
de espera, (3) llamar a \texttt{eval/run\_quality.py --role <rol>}.

\subsection{Motor de puntuación — \texttt{eval/scoring.py}}

El módulo \texttt{eval/scoring.py} implementa una \textbf{estrategia de comparación
multi-etapa} que gestiona la variabilidad en la codificación matemática de las
respuestas del modelo.

\paragraph{Extracción de respuesta por benchmark.}
\begin{itemize}
  \item \textbf{GSM8K (\texttt{eval/gsm8k.py}):} usa el patrón
    \texttt{\#\#\#\#\textbackslash{}s*(.*)} para extraer el valor numérico final.
    Se toma la \emph{última} ocurrencia del patrón en la respuesta, ya que los
    modelos a menudo reformulan la respuesta al final.
  \item \textbf{MATH-500 (\texttt{eval/math\_eval.py}):} la función
    \texttt{extract\_boxed\_answer} detecta la última expresión
    \texttt{\textbackslash{}boxed\{...\}} mediante seguimiento de profundidad
    de llaves, soportando construcciones anidadas como
    \texttt{\textbackslash{}boxed\{\textbackslash{}frac\{1\}\{2\}\}}.
\end{itemize}

\paragraph{Pipeline de comparación de tres etapas.}
La función \texttt{math\_answer\_match(prediction, reference)} de
\texttt{eval/scoring.py} implementa:

\begin{enumerate}
  \item \textbf{Normalización numérica} (\texttt{normalise\_numeric}):
    elimina signos \$, comas de millar, espacios y símbolo \%; intenta parsear
    como \texttt{float}. Si ambas respuestas son valores numéricos, compara con
    \texttt{math.isclose(rel\_tol=1e-4)}.
    Ejemplo: \texttt{"1,234.5"} $\to$ 1234.5; \texttt{"50\%"} $\to$ 0.5.

  \item \textbf{Comparación simbólica vía SymPy}
    (\texttt{\_sympy\_equal}): si la comparación numérica falla,
    \texttt{normalise\_math\_answer} convierte construcciones LaTeX a sintaxis
    SymPy (Tabla~\ref{tab:latex-sympy}), y llama a
    \texttt{sympy.simplify(p - r) == 0}.

  \item \textbf{Comparación exacta de strings}: fallback final para
    respuestas no numéricas.
\end{enumerate}

un diccionario con \texttt{total\_examples}, \texttt{scorable\_examples},

\section{Fase 2 — Destilación de Conocimiento}
\label{sec:distillation}

\textbf{Estado:} Generación completada (1819 prompts).
Training completado (jobs SLURM 37123021 y 37123022).
Evaluación post-destilación \emph{pendiente de ejecución}.

\subsection{Estrategia de destilación}

Se emplea \textbf{Knowledge Distillation black-box con etiquetas duras}
(\emph{hard-label KD}):

\begin{itemize}
  \item El teacher se accede \textbf{únicamente} a través de su endpoint de
    completado de texto (\texttt{/v1/completions} compatible con API OpenAI).
    No se accede a sus logits, no se usa divergencia KL, no hay etiquetas
    suaves (\textit{soft labels}). El término \emph{black-box} se refiere al
    teacher: sus pesos son inaccesibles; solo se consumen sus predicciones en texto.
  \item El estudiante se entrena por \textbf{SFT} (\textit{Supervised Fine-Tuning})
    sobre pares $(\text{prompt}, \text{completion\_teacher})$ minimizando la
    pérdida cross-entropy sobre los tokens de completado.
  \item Se usa \textbf{LoRA} (\textit{Low-Rank Adaptation}, PEFT) para mantener
    el número de parámetros entrenables bajo: solo las matrices de proyección de
    atención y las capas MLP se modifican mediante matrices de bajo rango, lo que
    reduce drásticamente el consumo de memoria y el riesgo de \emph{catastrophic
    forgetting}.
\end{itemize}

\paragraph{Parámetros LoRA: elección de $r$ y $\alpha$.}
El rango $r=64$ determina el rango de las matrices de adaptación:
$W_{\text{LoRA}} = W_0 + \frac{\alpha}{r} \cdot B A$
donde $A \in \mathbb{R}^{d \times r}$ y $B \in \mathbb{R}^{r \times d}$.
Con $\alpha=16$ y $r=64$, el factor de escala es $\alpha/r = 0.25$.
Los 7 módulos objetivo cubren toda la ruta de información: proyecciones de
atención (Q, K, V, O) y la red MLP (\texttt{gate\_proj}, \texttt{up\_proj},
\texttt{down\_proj}), maximizando la expresividad del adaptador.

\subsubsection{Script central — \texttt{distill/train\_student\_sft.py}}

La función \texttt{run(config\_path, student\_override, max\_samples)} implementa el
bucle completo de SFT en cinco etapas:

\paragraph{Etapa 1: Carga del modelo con distribución automática.}
\begin{lstlisting}[language=Python, caption={Carga del modelo estudiante con detección GPU.}]
has_gpu = torch.cuda.is_available()
use_bf16 = tcfg.get("bf16", True) and has_gpu and \
           torch.cuda.is_bf16_supported()
load_dtype = torch.bfloat16 if use_bf16 else torch.float32

model = AutoModelForCausalLM.from_pretrained(
    student_model,
    dtype=load_dtype,    # bfloat16 en H100, float32 en CPU
    device_map="auto",   # distribuye sobre las GPUs disponibles
)
\end{lstlisting}

\texttt{device\_map="auto"} hace que \texttt{accelerate} distribuya las capas
del modelo entre todas las GPUs disponibles del nodo, maximizando el uso de
VRAM. En el nodo de training (4$\times$H100-64\,GB), el 7B cabe en 1--2 GPUs,
y el 1.5B en 1 GPU; las restantes se usan para el batch en paralelo.

\paragraph{Etapa 2: Aplicación del adaptador LoRA.}
\begin{lstlisting}[language=Python, caption={Configuración e inicialización del adaptador LoRA.}]
peft_config = LoraConfig(
    r=64, lora_alpha=16, lora_dropout=0.05,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    bias="none",
    task_type=TaskType.CAUSAL_LM,
)
# CRITICO: habilitar gradientes en entradas ANTES de get_peft_model
# (requisito para que gradient_checkpointing funcione con PEFT)
model.enable_input_require_grads()
model = get_peft_model(model, peft_config)
model.print_trainable_parameters()
# -> trainable params: ~167M / 7B = ~2.4% (7B)
# -> trainable params:  ~29M / 1.5B = ~1.9% (1.5B)
\end{lstlisting}

\paragraph{Etapa 3: Tokenización con enmascaramiento de completado.}
La función \texttt{\_prepare\_dataset} carga el JSONL, filtra items con error,
tokeniza cada par $(\text{prompt}, \text{completion})$ y aplica el
\textbf{completion-only masking}: solo los tokens de la respuesta del teacher
contribuyen a la pérdida; los tokens del prompt reciben \texttt{label=-100}
(ignorados por \texttt{CrossEntropyLoss}).

\begin{lstlisting}[language=Python, caption={Completion-only masking en \_prepare\_dataset.}]
# Longitud del prompt en tokens
prompt_enc = tokeniser(prompt_text, add_special_tokens=True, truncation=False)
prompt_len = len(prompt_enc["input_ids"])

# Secuencia completa: prompt + completion
full_enc = tokeniser(
    prompt_text + completion_text,
    truncation=True, max_length=2048, padding="max_length",
)
input_ids     = full_enc["input_ids"]
attention_mask = full_enc["attention_mask"]

# Labels: -100 hasta prompt_len, id real para completion (no padding)
labels = [-100] * len(input_ids)
for i in range(prompt_len, len(input_ids)):
    if attention_mask[i] == 1:   # ignorar padding
        labels[i] = input_ids[i]
full_enc["labels"] = labels
\end{lstlisting}

Esta estrategia evita que el estudiante memorice la formulación del prompt y
fuerza el aprendizaje de las \emph{cadenas de razonamiento} del teacher.
El dataset final contiene 1819 ejemplos (filtrado de items sin completado o con
error), en formato \texttt{torch} con columnas \texttt{input\_ids},
\texttt{attention\_mask} y \texttt{labels}.

\paragraph{Etapa 4: Configuración de TrainingArguments.}
\begin{lstlisting}[language=Python, caption={Argumentos clave de entrenamiento.}]
training_args = TrainingArguments(
    output_dir=str(run_dir / "checkpoints"),
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,  # batch efectivo = 16
    learning_rate=2e-4,
    lr_scheduler_type="cosine",
    warmup_ratio=0.03,
    bf16=True,
    gradient_checkpointing=True,
    gradient_checkpointing_kwargs={"use_reentrant": False},
    eval_strategy="no",
    dataloader_num_workers=0,   # 0: evita deadlocks SLURM
    dataloader_pin_memory=False,
    seed=42,
)
\end{lstlisting}

\paragraph{Etapa 5: Entrenamiento y guardado del adaptador.}
\texttt{transformers.Trainer} gestiona el bucle de entrenamiento con
\texttt{DataCollatorForLanguageModeling(mlm=False)} (modelado de lenguaje
causal). Al finalizar las 3 épocas:
\begin{itemize}
  \item \texttt{trainer.save\_model(final\_dir)} guarda únicamente los pesos LoRA
    (no el modelo base) en \texttt{run\_dir/final\_adapter/}.
  \item \texttt{tokeniser.save\_pretrained(final\_dir)} guarda el tokenizador.
  \item Se escribe un \texttt{training\_manifest.json} con metadatos del run:
    modelo estudiante, ruta del adaptador, número de muestras, épocas, rango $r$.
\end{itemize}

\paragraph{[1/4] Fusión del adaptador LoRA con el modelo base.}
\begin{lstlisting}[language=Python, caption={Merge LoRA embebido en eval\_distilled\_7b.sbatch.}]
from peft import PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch, shutil, os

# Cargar modelo base en bfloat16
base = AutoModelForCausalLM.from_pretrained(
    'Qwen/Qwen2.5-7B-Instruct', torch_dtype=torch.bfloat16)
# Cargar adaptador LoRA sobre el base
model = PeftModel.from_pretrained(base, ADAPTER_DIR)
# Fusionar: absorbe los deltas LoRA en los pesos densos
merged = model.merge_and_unload()
# Guardar como HuggingFace estándar (sin dependencia de PEFT)
merged.save_pretrained(MERGED_DIR)
AutoTokenizer.from_pretrained('Qwen/Qwen2.5-7B-Instruct') \
             .save_pretrained(MERGED_DIR)
\end{lstlisting}

\noindent \texttt{merge\_and\_unload()} absorbe los deltas LoRA directamente en
los pesos del transformer: $W_{\text{merged}} = W_0 + \frac{\alpha}{r} B A$.
El resultado es un modelo HuggingFace estándar en
\texttt{results/distill/merged-qwen2.5-7b/}, directamente cargable por vLLM
sin ninguna dependencia de PEFT.

SERVER_PID=$!

\section{Reproducibilidad}
\label{sec:reproducibility}

Cada experimento es completamente reproducible. La reproducibilidad se garantiza
por diseño a múltiples niveles:

\paragraph{Nivel de configuración.}
Todos los hiperparámetros, rutas y opciones están externalizados en ficheros YAML
bajo \texttt{configs/}. Ningún hiperparámetro está hardcodeado en el código Python
salvo valores por defecto con \texttt{.get("param", default)}.

\paragraph{Nivel de ejecución.}
Cada run crea un directorio con timestamp UTC único, copia todos los YAMLs activos
y registra en \texttt{metadata.json}: hostname, GPU (nombre y memoria), versión CUDA,
versión Python, git SHA del repositorio y semilla aleatoria utilizada.

\paragraph{Nivel de aleatoriedad.}
La función \texttt{set\_seed(seed)} de \texttt{utils/reproducibility.py} fija las
semillas de Python \texttt{random}, NumPy y PyTorch (incluyendo
\texttt{torch.cuda.manual\_seed\_all}), garantizando determinismo en la inicialización
de LoRA y el shuffle del dataloader.

\paragraph{Nivel de generación.}
El teacher genera con \texttt{temperature=0.0} (decodificación greedy pura),
produciendo salidas determinísticas dado el mismo peso de modelo y prompt.
Todas las evaluaciones de calidad también usan decodificación greedy para
reproducibilidad exacta.

  metadata.json

\subsection{Objetivo final — Resultados}

El objetivo de la Fase~2 era \textbf{cerrar la brecha de calidad} de los estudiantes
preservando su ventaja de eficiencia. Los resultados de la evaluación post-destilación
(Tabla~\ref{tab:postdistill}) muestran que la destilación ha sido efectiva:

\begin{itemize}
  \item El \textbf{student 1.5B} mejora significativamente: $+10.64$ puntos en MATH-500
    (de 36.81\% a 47.45\%), acercándose al teacher (70.51\%) y validando la hipótesis

  \item El \textbf{student 7B} muestra un trade-off: pierde en aritmética simple

    sugiriendo que la destilación refuerza capacidades de razonamiento complejo.
\end{itemize}

La ventaja de eficiencia se preserva: los modelos destilados mantienen sus ganancias
de $2.15\times$ y $4.14\times$ en throughput, respectivamente. La destilación ha
cumplido su objetivo de mejorar la calidad sin sacrificar rendimiento.
